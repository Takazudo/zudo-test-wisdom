## テストレベルが重要な理由

AIエージェントがバグを修正して解決済みと宣言する際、その信頼性は検証に使用したテスト手法に依存します。ユニットテストはロジックの正しさを確認できますが、ユーザーが実際に画面上で結果を見られるかどうかは確認できません。「ロジック的に正しい」と「視覚的に正しい」の間のギャップが、AI支援開発における誤った確信の最も一般的な原因です。

このガイドでは、カバレッジが増加しブラインドスポットが減少する**5つのテストレベル**を定義します。核となる原則はシンプルです：

<Note>
テストレベルを、実際に検証している対象に合わせてください。ロジックを確認するならレベル1で十分です。画面上に何かが表示されているかを確認するなら、レベル5が必要です。
</Note>

## このガイドの対象者

このガイドは2つの対象者に向けて書かれています：

- 自身の作業を検証する際に適切なテストアプローチを選択する必要がある**AIエージェント**（Claude Code、Copilotなど）
- AIエージェントと協業し、テストエスカレーションラダーを理解したい**開発者**

## エスカレーションの原則

テストは繰り返すのではなく、エスカレーションすべきです。下位レベルのテストがパスしても問題が解消しない場合は、次のレベルに進みます：

1. **レベル1** -- ユニット/ロジックテスト（vitest、jest）
2. **レベル2** -- DOMベースのコンポーネントテスト（jsdom、Testing Library）
3. **レベル3** -- ビルド出力検証（ビルドされたファイルを読む）
4. **レベル4** -- E2Eブラウザテスト（Playwright、headless browser）
5. **レベル5** -- 決定論的 + 視覚的検証（verify-ui + headless browser）

各レベルは、下位レベルでは検出できないブラインドスポットをカバーします。よくある失敗パターンは、バグがレベル4や5を必要としているのにレベル1にとどまることです。

## このガイドの使い方

1. まず[判断ガイド](/pj/zudo-test-wisdom/ja/docs/decision-guide)で、現在のタスクに適したレベルを判断します
2. [テストレベル](/pj/zudo-test-wisdom/ja/docs/testing-levels)セクションで各レベルの詳細を確認します
3. [実践パターン](/pj/zudo-test-wisdom/ja/docs/real-world-patterns)で本番環境で検証済みの設定を参照します
4. [ツールリファレンス](/pj/zudo-test-wisdom/ja/docs/tools-reference)をクイックルックアップ用に開いておきます
